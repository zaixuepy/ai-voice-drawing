# PRD：AI 语音绘图工具
**版本：** v1.0 | **作者：** 张宇翔 邓浩文 | **日期：** 2026-06-12 | **状态：** Draft

---

## 1. 执行摘要

这是一款纯语音控制的绘图工具，用户无需使用鼠标或键盘，仅通过说话即可在画布上创建、修改、连接图形元素。

**核心差异化：** 市面上所有同类工具（Canva AI、Excalidraw+AI开源项目）均假设用户知道自己想要什么，只是将文字输入替换为语音。本产品真正解决的是"用户无法精确描述意图"的问题——通过上下文理解、模糊指令容错和主动确认机制，让AI比用户更早理解用户想画什么。

**一句话定位：** 用说话代替鼠标，把脑子里的图画出来。

---

## 2. 问题陈述

### I am / Trying to / But / Because 框架

**I am** 一个需要快速将脑中结构性想法可视化的人（产品经理、学生、创作者）

**Trying to** 把我脑子里的流程、结构、关系画成一张图

**But** 我不知道从哪里开始，画图工具操作繁琐，而且当我说"感觉不好看"或"再清晰一点"时，没有任何工具能理解我的意思

**Because** 现有工具要么需要精确的操作指令（鼠标点击），要么需要精确的文字描述（prompt），都假设用户已经知道自己要什么

### 问题严重程度

- 语音绘图领域目前**无成熟商业产品**，仅有Hackathon原型
- Excalidraw + AI开源生态活跃（2026年2-6月持续更新），但**全部为文字输入，无语音接入**
- Canva AI 2.0已支持语音输入，但本质是"语音prompt→AI生成完整设计"，**不是逐步构建和修改**

## 3. 目标用户

### 主要用户（MVP阶段聚焦）

**产品经理 / 技术同学**
- 场景：需要快速画流程图、架构图、用户旅程图
- 痛点：有想法但操作工具耗时，思维被打断
- 为什么语音是刚需：思维流动时不想切换到鼠标操作模式

### 次要用户（后续迭代）

- 学生：整理笔记、梳理知识结构
- 创作者：快速草图头脑风暴
- 教师：课堂边讲边画示意图

### 明确排除的用户

- 需要精细视觉设计的用户（配色、海报）→ 属于方向B，本MVP不做
- 肢体障碍人士的无障碍需求 → 市场太小，不作为主打方向

## 4. 战略背景

### 为什么现在做

- Excalidraw开源生态在2026年Q1-Q2爆发式增长，MCP工具链成熟，技术门槛大幅降低
- Web Speech API已在主流浏览器稳定支持，STT成本趋近于零
- 语音绘图是**确认的市场空白**：竞品调研显示无任何成熟商业产品

### 竞品现状

| 产品 | 语音能力 | 核心缺陷 |
|------|---------|---------|
| Canva AI 2.0 | 语音→生成完整设计 | 不支持逐步构建和修改 |
| Adobe Firefly | 无语音绘图能力 | 完全空白 |
| Excalidraw开源生态 | 无 | 全部文字输入 |
| VocalCanvas (Hackathon) | 有 | 原型阶段，无上下文理解 |

### 不做会怎样

语音绘图窗口期不超过12个月，Canva、Notion等产品线会逐步延伸至此。现在是小团队建立先发优势的时机。

## 5. 解决方案概述

### 我们做什么

一个Web应用，分为两个区域：

```
左侧：对话区
- 麦克风按钮（核心入口）
- 实时语音转文字显示
- AI回应和确认信息

右侧：画布区
- 嵌入Excalidraw
- 实时反映语音指令结果
- 用户可看到图形逐步生成
```

### 核心体验流程

```
用户点击麦克风 → 说一句话 → 文字出现在对话区
→ AI解析指令 → 画布实时更新 → 用户看到结果
→ 继续说下一句（可引用上文："把那个移到右边"）
```

### 两层设计（核心产品洞察）

**第一层：意图澄清**
当用户说模糊的话（"感觉不好看"、"再清晰一点"），AI不乱猜，而是用一句话反问确认方向，引导用户逐步明确意图。

**第二层：精准执行**
意图明确后，AI将语音转化为结构化JSON指令，精准调用Excalidraw API执行。

### 我们不做什么（Out of Scope）

- ❌ 图像生成（Stable Diffusion类）
- ❌ 配色和视觉美化
- ❌ 实时会议监听（持续流式token消耗不可控）
- ❌ 飞书/钉钉集成（72小时内不做）
- ❌ 移动端适配
- ❌ 多人协作

## 6. 成功指标

### 北极星指标

**语音指令成功执行率** > 80%（用户说一句话，画布正确响应）

### 护栏指标

- 简单指令端到端延迟 < 1.5秒
- 复杂指令端到端延迟 < 3秒
- 单次对话平均token消耗 < 1000 tokens

### Demo验收标准

评审能在Demo中看到：
1. 用户说话 → 画布出现图形（基础闭环）
2. 用户说"把那个变成红色" → AI理解上下文，正确修改（上下文理解）
3. 用户说"感觉太乱了" → AI反问引导而非乱猜（容错机制）

## 7. 用户故事与需求

### US-001 基础绘图（P0，MVP必须）

**As a** 用户
**I want to** 说"画一个矩形"然后看到画布上出现矩形
**So that** 我能用最简单的方式验证语音绘图是否工作

**验收标准：**
- Given 用户点击麦克风按钮
- When 用户说"画一个矩形"/"画一个圆"/"画一个箭头"
- Then 画布在3秒内出现对应图形

---

### US-002 属性设定（P0，MVP必须）

**As a** 用户
**I want to** 说"画一个红色的大圆在左上角"
**So that** 我能一句话指定图形的颜色、大小和位置

**验收标准：**
- Given 画布为空
- When 用户说包含颜色/大小/位置描述的指令
- Then 图形属性与描述一致（颜色准确，位置大致正确）

---

### US-003 上下文引用（P0，MVP必须）

**As a** 用户
**I want to** 说"把那个圆变成蓝色"
**So that** 我不需要每次都完整描述目标元素

**验收标准：**
- Given 画布上已有一个红色圆
- When 用户说"把那个圆变成蓝色"
- Then AI正确识别目标元素并修改颜色

---

### US-004 连接图形（P1，核心功能）

**As a** 用户
**I want to** 说"用箭头连接左边的圆和右边的方块"
**So that** 我能表达元素之间的关系

**验收标准：**
- Given 画布上有两个以上图形
- When 用户说连接指令
- Then 画布出现箭头连接对应图形

---

### US-005 模糊指令容错（P1，差异化亮点）

**As a** 用户
**I want to** 说"感觉不够清晰"时AI能引导我
**So that** 我不需要知道专业术语也能修改图

**验收标准：**
- Given 画布上有若干图形
- When 用户说模糊的评价性语言
- Then AI用一句话反问（"是元素太密集，还是颜色太相近？"）而不是乱猜执行

---

### US-006 添加文字标注（P1）

**As a** 用户
**I want to** 说"在那个方块里写'用户登录'"
**So that** 我的图有清晰的文字说明

---

### US-007 撤销操作（P1）

**As a** 用户
**I want to** 说"撤销"或"不对，重来"
**So that** 我能纠正错误操作

## 8. 优先级排序

### KANO模型分析

| 功能 | KANO类型 | 优先级 | 理由 |
|------|---------|------|------|
| 语音→画基础图形 | 基本型（Must-have） | P0 | 没有这个就不是这个产品 |
| 上下文引用（"那个"） | 期望型（Performance） | P0 | 直接决定体验好不好 |
| 属性设定（颜色/大小） | 期望型 | P0 | 用户期待能一句话说清楚 |
| 连接图形 | 期望型 | P1 | 流程图必须，但可以第二天做 |
| 模糊指令容错 | 兴奋型（Delighter） | P1 | 评审最惊喜的点，差异化核心 |
| 文字标注 | 期望型 | P1 | 实用但不是第一天必须 |
| 撤销操作 | 基本型 | P1 | 必须有但实现简单 |
| 复杂指令拆解 | 兴奋型 | P2 | 迭代再做 |
| 导出PNG/SVG | 基本型 | P2 | Demo阶段不是重点 |
| 飞书/钉钉集成 | 兴奋型 | ❌ | 72小时内不做 |

```
1. STT使用Web Speech API → 零成本
2. 按句触发API，非持续流式 → 可控
3. 画布状态压缩传输 → 减少token
4. 简单指令（撤销/清空）本地处理 → 不调API
5. 目标：每次对话平均<1000 tokens
```

### 延迟控制策略（⚠️ 需反复确认）

```
1. STT实时显示转写文字（用户感知到在处理）
2. Claude API使用streaming输出
3. 画布操作在第一个token返回时开始执行
4. 目标：简单指令<1.5秒，复杂指令<3秒
```

## 附录D：技术架构

```
前端：React（单页面应用）
画布：Excalidraw（npm包嵌入）
语音：Web Speech API（浏览器原生）
AI：Claude API（claude-sonnet）
部署：Vercel

核心数据流：
用户说话
  → Web Speech API → 文字
  → 画布状态序列化（压缩）
  → Claude API（系统prompt + 画布状态 + 用户指令）
  → JSON指令返回
  → Excalidraw API执行
  → 画布更新
```

**系统Prompt结构（Agent3核心产出）：**
```
角色定义：你是绘图指令解析器
输出格式：严格JSON，包含actions数组
画布状态：当前所有元素的简化描述
容错规则：模糊指令时返回clarify类型而非执行
上下文：保留最近5轮对话历史
```

**JSON指令格式示例：**
```json
{
  "actions": [
    {"type": "add", "shape": "rect", "x": 100, "y": 100, "width": 120, "height": 60, "color": "#4A90D9", "label": "用户登录"},
    {"type": "move", "target": "circle_001", "x": 300, "y": 200},
    {"type": "connect", "from": "rect_001", "to": "rect_002"}
  ]
}

// 模糊指令时返回：
{
  "actions": [],
  "clarify": "是元素太密集，还是颜色太相近？"
}
```
